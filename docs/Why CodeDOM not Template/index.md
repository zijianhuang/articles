# Why I have chosen CodeDOM rater than Roslyn in 2015

The following content is generated by ChatGPT which has proactively suggested when talking about Roslyn vs template vs CodeDOM. In 2015, I am aware of Roslyn and had fairly decent experiences in using various template based code generation libraries.

There is a specific architecture where CodeDOM still shines:
**Generating multiple language clients from a single .NET metadata model**, especially when the server is **ASP.NET Web API / ASP.NET Core**.

This pattern was used by tools like **WebApiClientGen**.

The key idea is:

```
.NET reflection
      ↓
language-neutral CodeDOM tree
      ↓
multiple language providers
```

This avoids **duplicating generation logic for each language**, which is the main weakness of template systems.

---

# 1. Problem templates struggle with

Suppose you want to generate API clients for:

* C#
* TypeScript
* Java
* Python

A template approach looks like:

```
OpenAPI model
     ↓
C# template
TS template
Java template
Python template
```

Each template must implement:

* type mapping
* parameter handling
* async handling
* serialization rules

This leads to **duplicated logic across templates**.

---

# 2. CodeDOM architecture

A CodeDOM-based generator separates **logic from syntax**.

Architecture:

```
ASP.NET Controllers
        ↓
Reflection
        ↓
API Metadata Model
        ↓
CodeDOM tree (language neutral)
        ↓
C# provider
TypeScript provider
VB provider
```

Only the **syntax layer changes**.

---

# 3. Example pipeline

### Step 1 — Reflect controllers

```
public class OrdersController : ControllerBase
{
    [HttpGet]
    public Task<Order[]> GetOrders(int customerId);
}
```

Reflection extracts:

```
Controller
  name: Orders
  method: GetOrders
  return: Task<Order[]>
  parameter: int customerId
```

---

### Step 2 — Build CodeDOM model

Create a **client method representation**:

```
CodeMemberMethod
    Name: GetOrders
    ReturnType: Task<Order[]>
    Parameter: int customerId
```

Add HTTP logic:

```
GET /api/orders?customerId={customerId}
```

---

### Step 3 — Generate languages

The same CodeDOM tree produces:

### C#

```csharp
public Task<Order[]> GetOrders(int customerId)
{
    return http.GetAsync<Order[]>($"api/orders?customerId={customerId}");
}
```

### TypeScript

```ts
getOrders(customerId: number): Promise<Order[]> {
    return this.http.get<Order[]>(`api/orders?customerId=${customerId}`);
}
```

### VB.NET

```vb
Public Function GetOrders(customerId As Integer) As Task(Of Order())
    Return http.GetAsync(Of Order())($"api/orders?customerId={customerId}")
End Function
```

The **logic is identical**.

---

# 4. Why CodeDOM beats templates here

### 1️⃣ Shared generation logic

Templates require:

```
method.mustache
method.ts.mustache
method.cs.mustache
```

CodeDOM uses:

```
method logic once
syntax generated by provider
```

---

### 2️⃣ Strong typing during generation

CodeDOM objects:

```
CodeTypeReference
CodeMemberMethod
CodeParameterDeclarationExpression
```

This avoids bugs like:

```
List<Order>
List<Order
List<Order>>
```

Templates are **string manipulation**.

---

### 3️⃣ Language transformation layer

The provider can convert constructs.

Example mapping:

| CodeDOM          | C#               | TypeScript   |
| ---------------- | ---------------- | ------------ |
| `Task<T>`        | `Task<T>`        | `Promise<T>` |
| `DateTime`       | `DateTime`       | `Date`       |
| `IEnumerable<T>` | `IEnumerable<T>` | `T[]`        |

Only the **provider implements mapping**.

---

### 4️⃣ Easier maintenance

Without CodeDOM:

```
50 templates
```

With CodeDOM:

```
1 generator
+ few providers
```

Modern **industrial-grade code generators** (especially API SDK generators) often use a **three-layer architecture** that combines the strengths of CodeDOM and AST systems while avoiding their weaknesses.

The pattern is:

```
Metadata → Intermediate Semantic Model → Language AST → Source Code
```

This architecture is used by large generators such as
OpenAPI Generator and AutoRest.

---

# 1. The three-layer architecture

## Layer 1 — Metadata extraction

Source information is collected from:

* OpenAPI / Swagger
* .NET reflection
* GraphQL schema
* database schema

Example:

```
GET /orders
parameter: customerId:int
return: Order[]
```

Converted into a **language-neutral model**.

Example structure:

```
ApiEndpoint
  name: GetOrders
  httpMethod: GET
  route: /orders
  parameters:
      customerId:int
  returnType: Order[]
```

This layer contains **no language information**.

---

# 2. Layer 2 — Intermediate Semantic Model (ISM)

This is the **core of modern generators**.

It describes the **program structure**, not the syntax.

Example model:

```
Class
  name: OrdersClient
  methods:
      Method
         name: GetOrders
         parameters:
             customerId:int
         returnType: List<Order>
```

This layer defines:

* types
* classes
* methods
* properties
* generics
* async semantics

But **no language syntax**.

Example semantics:

```
AsyncResult<List<Order>>
```

Later translated to:

| Language   | Result                           |
| ---------- | -------------------------------- |
| C#         | `Task<List<Order>>`              |
| TypeScript | `Promise<Order[]>`               |
| Java       | `CompletableFuture<List<Order>>` |

---

# 3. Layer 3 — Language AST

Each language has its own **Abstract Syntax Tree**.

Examples:

### C#

AST using Roslyn:

```
CompilationUnit
  Namespace
    Class
      Method
```

Libraries:

* `Microsoft.CodeAnalysis`

---

### TypeScript

AST using the TypeScript compiler API:

```
SourceFile
  ClassDeclaration
    MethodDeclaration
```

---

### Java

AST using tools like:

* JavaPoet

---

# 4. Code generation pipeline

The full pipeline becomes:

```
           OpenAPI
              │
              ▼
      Metadata Extraction
              │
              ▼
     Intermediate Semantic Model
              │
     ┌────────┼─────────┐
     ▼        ▼         ▼
 C# AST    TS AST    Java AST
     ▼        ▼         ▼
  C# code  TS code   Java code
```

---

# 5. Why this architecture works better

## 1️⃣ Clean separation of concerns

| Layer          | Responsibility           |
| -------------- | ------------------------ |
| Metadata       | extract information      |
| Semantic model | describe program meaning |
| AST            | produce correct syntax   |

Each layer is independent.

---

## 2️⃣ No language leakage

CodeDOM failed because language differences leaked into the model.

Example:

```
Task<T>
Promise<T>
CompletableFuture<T>
```

Instead the model uses:

```
Async<T>
```

Then each language converts it.

---

## 3️⃣ Full language power

AST allows:

* decorators
* annotations
* async patterns
* advanced generics
* comments
* formatting

CodeDOM cannot represent these.

---

## 4️⃣ Safe code generation

AST ensures **syntactically correct code**.

Templates cannot guarantee this.

Example template bug:

```
List<Order
```

AST prevents invalid structures.

# 7. Why big generators use this

Large SDK generators must support **many languages**.

Example tools support:

| Tool              | Languages |
| ----------------- | --------- |
| OpenAPI Generator | 50+       |
| AutoRest          | 10+       |

Maintaining **templates only** becomes extremely difficult.

The **semantic model + AST architecture scales much better**.

---

# 8. Relationship to CodeDOM

CodeDOM attempted to do:

```
Semantic model + AST
```

in **one layer**, which made it too limited.

Modern architecture **splits them**.

```
Semantic Model
    +
Language AST
```

This solves the design problem.
